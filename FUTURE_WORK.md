# 미래 개발 제안: Solidity 파싱 및 스크립트 생성 고도화

이 문서는 `forge-scriptgen`을 실무 수준으로 확장하기 위해 고려해야 할 후속 과제들을 정리합니다. 각 항목은 **왜 필요한지(Why)**, **어떻게 구현할지(How)**, **완료 시 기대 결과(Outcome)**를 명시해 우선순위 선정과 작업 계획 수립에 도움을 주도록 구성했습니다.

---

## 1. AST 기반 Solidity 파싱 도입
- **Why**
  - 현재 파서는 문자열 검색과 괄호 추적에 의존하기 때문에 새로운 솔리디티 문법(커스텀 오류, 모디파이어, using for, multi-line 선언, 상속 체인 등)이 등장하면 쉽게 무너질 위험이 있습니다 (`main.rs:347-515`).
  - 실무 코드베이스는 `contract` 뿐 아니라 `abstract contract`, `library`, `interface`, `using` 블록이 얽혀 있고, 주석/문자열 속의 키워드가 혼재되어 있어 정규식 기반 파싱으로는 장기적으로 유지보수가 어렵습니다.
- **How**
  1. 후보 라이브러리 조사: `solang`, `ethers-solc`, `foundry`의 AST 출력 등에서 JSON AST를 얻을 수 있는지 조사합니다.
  2. 최소 작업 단위를 정의: "파일 ➜ AST ➜ ContractInfo" 변환 파이프라인을 별도 모듈로 분리하고, 기존 파서를 점진적으로 치환합니다.
  3. 테스트 세트 확장: 다양한 계약 패턴(상속, 추상, 다중 생성자, 매크로)으로 회귀 테스트를 구성합니다.
  4. AST 파싱 결과를 `ContractInfo`에 매핑하고, 기존 문자열 파서 대비 성능/정확도를 검증합니다.
- **Outcome**
  - 복잡한 솔리디티 문법에도 안정적으로 대응할 수 있는 계약/생성자 정보 추출 기능 확보.
  - 유지보수 비용 감소 및 실무 코드베이스에 적용 가능.

## 2. 파싱 백엔드 추상화 계층 도입
- **Why**
  - AST 파서 도입 전환 중에도 기존 문자열 파서를 완전히 제거하지 않고 비교/폴백 전략을 유지할 수 있어야 합니다.
  - 테스트 및 디버깅 시 파서 교체가 간편해야 하며, CLI 플래그로 선택하도록 하면 사용자 실험이 수월합니다.
- **How**
  1. `trait ContractParser` 정의: `fn parse(&self, source: &str) -> Result<Vec<ContractInfo>>` 형태.
  2. 현재 문자열 파서를 `StringWalkerParser`로 캡슐화하고, 추후 `AstParser`를 추가.
  3. `discover_contracts`가 특정 파서를 주입받도록 리팩터링(예: 환경 변수/플래그 기반 선택).
  4. 통합 테스트에서 두 파서를 모두 돌려 결과를 비교하도록 수정.
- **Outcome**
  - 파서 구현 교체/확장 시 코드 변경 범위를 최소화.
  - 베타 기능(예: AST 파서)을 안전하게 실험하고, 안정화되면 기본값으로 승격 시키는 경로 확보.

## 3. 생성자 메타데이터 확장
- **Why**
  - 현재 생성자 파라미터는 문자열 그대로 출력합니다 (`main.rs:794-802`). 타입, 기본값, `indexed`, `memory` 등 상세 정보를 활용하지 못합니다.
  - 복잡한 구조체/튜플 타입은 사용자가 직접 literal을 입력해야 하므로 실수가 잦습니다.
- **How**
  1. AST에서 파라미터 타입/네임스페이스 정보를 추출해 `ConstructorParam`에 구조화된 필드를 추가합니다.
  2. 구조체/배열 타입에 대해 템플릿 예시를 CLI에 제공하거나, JSON 입력 검증 로직을 강화합니다.
  3. 향후 자동 타입 변환(예: JSON ➜ Solidity literal) 기능을 염두에 둔 설계.
- **Outcome**
  - 사용자 입력 검증 강화 및 스크립트 생성 시 타입 안전성 향상.
  - 향후 자동화(예: JSON→Solidity literal 변환) 기반 마련.

## 4. 보안 친화적 비공개 키 처리
- **Why**
  - 현재는 프라이빗 키를 평문으로 스크립트에 박아 넣습니다 (`main.rs:781-812`). 실무에서는 키를 코드에 남기지 않고 환경변수/키스토어로 관리합니다.
- **How**
  1. `--private-key-env <VAR>` 옵션 등으로 환경변수를 참조하도록 스크립트 템플릿을 변경.
  2. `.env` 또는 Foundry의 `cast wallet` 연동 가이드를 문서에 추가.
  3. 템플릿 생성 시 `vm.envUint("VAR")` 형태로 변환하도록 선택지를 제공합니다.
- **Outcome**
  - 생성된 스크립트가 보안 모범 사례를 따르도록 개선.
  - 팀 차원에서 스크립트를 바로 Commit/CI에 올려도 키 노출 위험 없음.

## 5. 템플릿 시스템 도입 및 커스터마이징 지원
- **Why**
  - 현재 스크립트 포맷은 하드코딩된 하나의 문자열입니다 (`main.rs:804-812`). 프로젝트마다 로깅, 이벤트, 멀티컨트랙트 배포 등 요구사항이 다릅니다.
- **How**
  1. 템플릿 엔진(예: `handlebars`, `tera`) 또는 `indoc!` 매크로를 활용해 템플릿 파일을 외부 리소스로 분리.
  2. 사용자 정의 템플릿 경로를 CLI로 받는 기능 추가.
  3. 템플릿 내에서 `{constructor.call}`, `{broadcast.start}` 등 명시적인 치환 토큰을 정의.
- **Outcome**
  - 다양한 배포 시나리오를 유연하게 지원.
  - 유지보수 시 문자열 포맷팅 버그를 줄이고, PR 리뷰 가독성 향상.

## 6. 고급 CLI/UX 개선
- **Why**
  - 현재 CLI는 직접 파싱 로직을 구현해 기능 추가 시 분기문이 계속 늘어날 우려가 있습니다 (`main.rs:171-259`).
  - 사용자가 오입력했을 때 가능한 옵션을 제안하거나, 자동 완성/인터랙티브 리스트를 제공하면 생산성이 높아질 수 있습니다.
- **How**
  1. `clap`/`clap_complete` 도입으로 선언적 CLI 정의, 셸 자동 완성 파일 생성.
  2. `--interactive` 모드를 추가해 계약 목록을 메뉴 형태로 선택하도록 구성(TUI 라이브러리 검토).
  3. 에러 메시지를 구조화해 어떤 옵션이 잘못되었는지 명확히 안내.
- **Outcome**
  - 사용자 경험 개선 및 문서화 부담 감소.
  - CLI 옵션 확장 시 코드 유지보수 용이.

---

### 우선순위 제안
1. **AST 기반 파싱 + 파서 추상화**: 정확도와 유지보수성을 동시에 해결하므로 1차 목표로 추천.
2. **보안 강화(프라이빗 키 처리)**: 실무 적용 전 필수 안전장치.
3. **템플릿 시스템 도입**: 팀별 스크립트 커스터마이징 수요가 있을 경우 빠르게 도입.
4. **생성자 메타데이터 확장**: AST 도입 후 자연스럽게 연결되는 중장기 과제.
5. **고급 CLI/UX**: 사용자 피드백 기반으로 순차 개선.

각 작업은 독립적으로 진행할 수 있으나, 1→2→5→3→6 순으로 추진하면 코드 베이스의 안정성을 유지하면서 기능 면에서도 균형 있게 발전시킬 수 있습니다.
